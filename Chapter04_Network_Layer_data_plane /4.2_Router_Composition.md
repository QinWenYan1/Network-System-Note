## ✅ 知识点1: 路由器体系架构概览

**理论**

高层面（非常简化的）通用路由器体系架构包含两个核心功能：

- **路由(Routing)**：运行**路由选择算法/协议**（RIP、OSPF、BGP）生成**路由表**
  - 属于**控制平面**
  - 以**软件**实现，运行在**毫秒级**时间尺度
- **转发(Forwarding)**：从输入到输出链路交换数据报——根据路由表进行分组的**转发**
  - 属于**数据平面**
  - 以**硬件**实现，运行在**纳秒级**时间尺度

**架构组成**
- **路由处理器(routing processor)**：计算转发表，然后下载到输入端口
- **高速交换结构(high-speed switching fabric)**：核心组件，负责将分组从输入端口移到输出端口
- **路由器输入/输出端口**：分组进入和离开路由器的物理接口

**注意点**
- ⚠️ **速度鸿沟**：控制平面（路由）运行在毫秒级；数据平面（转发）运行在纳秒级——相差**百万倍**
- 💡 **理解技巧**：路由处理器像餐厅经理决定菜单（路由表）；交换结构像厨房流水线实际出餐（转发）
- 🔄 **知识关联**：转发表是控制平面和数据平面之间的桥梁

## ✅ 知识点2: 输入端口功能

**理论**

输入端口执行三层处理：

1. **物理层**：比特级接收（`线路端接/line termination`）
2. **数据链路层**：链路层协议动作、解封装（如**以太网/Ethernet**）
3. **分布式交换**：查表和转发决策

**分布式交换**
- 根据数据报头部信息（如**目的地址**），输入端口在本地内存中的**转发表**里查找合适的输出端口（**匹配+行动/match + action**）
- 两种转发范式：
  - **基于目标的转发(destination-based forwarding)**：仅仅依赖于IP数据报的**目标IP地址**（传统方法）
  - **通用转发(generalized forwarding)**：基于**头部字段的任意集合**（SDN方法）

**注意点**
- ⚠️ **本地查表**：转发表查找以**分布式**方式在每个输入端口本地完成，不是集中式
- 📋 **术语提醒**：`line termination` = 物理层信号接收；`decapsulation` = 去除链路层头部
- 🔄 **知识关联**：输入端口的链路层处理运行在**毫秒时间框架**，但转发决策必须快得多

## ✅ 知识点3: 基于目标的转发与最长前缀匹配

**理论**

**转发表（传统方式）**
- 映射：**目的地址范围** → **链路接口**
- 示例表：

| 目的地址范围 | 链路接口 |
|:---|:---:|
| `11001000 00010111 00010000 00000000` 至 `11001000 00010111 00010111 11111111` | 0 |
| `11001000 00010111 00011000 00000000` 至 `11001000 00010111 00011000 11111111` | 1 |
| `11001000 00010111 00011001 00000000` 至 `11001000 00010111 00011111 11111111` | 2 |
| 其他 | 3 |

**问题**：如果地址范围**没有划分得特别规整**，会发生什么？

**解决方案：最长前缀匹配(longest prefix matching)**
- 当给定目标地址查找转发表时，采用**最长地址前缀匹配**的目标地址表项
- 前缀表示例：

| 目的地址范围 | 链路接口 |
|:---|:---:|
| `11001000 00010111 00010*** ********` | 0 |
| `11001000 00010111 00011000 ********` | 1 |
| `11001000 00010111 00011*** ********` | 2 |
| 其他 | 3 |

- 查找示例：
  - 目的地址：`11001000 00010111 00010` → 匹配接口 **0**（最长前缀）
  - 目的地址：`11001000 00010111 00011000` → 匹配接口 **1**（比接口2更具体）

**硬件实现：TCAM**
- **TCAMs**（Ternary Content Addressable Memories，三态内容寻址存储器）：执行最长前缀匹配的硬件
- **内容可寻址**：将地址交给TCAM，它可以在**一个时钟周期内**检索出地址，不管表空间有多大
- Cisco Catalyst系列路由器：在TCAM中可以存储多达约**100万条路由表项**

**注意点**
- ⚠️ **为什么是最长前缀？** 更具体的路由（更长前缀）应该优先于更不具体的路由——就像包裹上更精确的地址
- 💡 **理解技巧**：最长前缀匹配就像找最精确的邮编匹配——"10001"比"1000*"更具体
- 📋 **术语提醒**：TCAM = 三态内容寻址存储器；prefix matching = 前缀匹配

## ✅ 知识点4: 输入端口缓存与头端阻塞

**理论**

- 当**交换机构的速率**小于**输入端口的汇聚速率**时，分组需要在输入端口**排队**
- 后果：
  - **排队延迟**
  - **由于输入缓存溢出造成丢失**

**头端阻塞(Head-of-the-Line, HOL Blocking)**
- 定义：排在**队头的数据报**阻止了队列中其他数据报向前移动
- 场景：
  - **输出端口竞争**：多个分组发往同一个输出端口——只能传递一个，其他被阻塞
  - **交叉流量延迟**：绿色分组排在被阻塞的红色分组后面，即使它自己的输出端口是空闲的，也无法前进

**注意点**
- ⚠️ **HOL是主要瓶颈**：即使交换结构很快，头端阻塞也会限制路由器吞吐量
- 💡 **理解技巧**：HOL阻塞就像收费站前面有人忘带钱包——即使你的车道技术上开着，你也得等他处理完才能走
- 🔄 **知识关联**：HOL阻塞特指发生在**输入排队交换机**中，输出排队或纵横结构不会出现

## ✅ 知识点5: 交换结构

**理论**

**功能**：将分组从输入缓冲区传输到合适的输出端口。

**交换速率**：分组可以按照该速率从输入传输到输出。
- N个输入端口时，理想情况下交换机构的交换速度是输入线路速度的**N倍**，才不会成为瓶颈。

**三种典型的交换机构：**

**1. 通过内存交换（第一代路由器）**
- 在**CPU直接控制**下的交换，采用传统的计算机架构
- 分组被拷贝到**系统内存**，CPU从分组的头部提取出目标地址，查找转发表，找到对应的输出端口，拷贝到输出端口
- **限制**：
  - 转发速率被**内存的带宽**限制（数据报通过总线**两遍**）
  - 一次只能转发**一个分组**

**2. 通过总线交换**
- 数据报通过**共享总线**，从输入端口转发到输出端口
- **总线竞争**：交换速度受限于**总线带宽**
- 一次只能处理**一个分组**
- 实例：1 Gbps总线（Cisco 1900）；32 Gbps总线（Cisco 5600）
- 对于接入或企业级路由器速度足够，但**不适合区域或骨干网络**

**3. 通过互联网络交换（如crossbar纵横结构）**
- **同时并发转发多个分组**，克服总线带宽限制
- 类型：**Banyan（榕树）网络**、**crossbar（纵横）**和其他的互联网络
- 机制：当分组从端口A到达，转给端口Y；控制器短接相应的两个总线
- 高级设计：将数据报分片为固定长度的**信元(cell)**，通过交换网络交换
- 实例：**Cisco 12000**：以**60Gbps**的交换速率通过互联网络

**注意点**
- ⚠️ **演进路线**：内存 → 总线 → 纵横，每一代消除一个瓶颈
- 💡 **理解技巧**：内存交换像单车道桥；总线交换像单条高速公路；纵横结构像全互通立交桥，多车道同时通行
- 📋 **术语提醒**：Crossbar = 纵横交换结构；Banyan = 榕树网络；cell = 信元

## ✅ 知识点6: 输出端口、缓存与调度机制

**理论**

**输出端口功能**
- 当数据报从交换机构的到达速度比**传输速率快**时，需要**输出端口缓存**
- 数据报（分组）可能会被丢弃，由于拥塞，**缓冲区没有空间**
- 由**调度规则**选择排队的数据报进行传输
- 问题：**优先权调度**——谁会获得最优性能？**网络中立？**

**输出端口排队**
- 假设交换速率 $R_{switch}$ 是 $R_{line}$ 的**N倍**（N：输入端口的数量）
- 当多个输入端口同时向输出端口发送时，缓冲该分组
- **排队带来延迟**；**由于输出端口缓存溢出则丢弃数据报**

**需要多少缓存？**
- **RFC 3439 拇指规则（经验性规则）**：平均缓存大小 = 典型的**RTT**（例如：250ms）× **链路容量C**
  - 例：C = 10 Gbps链路，RTT = 250ms → 缓存 = 250ms × 10Gbps = **2.5 Gbit**
- **最近的一些推荐**：有**N（非常大）**个流时，缓存大小等于 $\frac{RTT \cdot C}{\sqrt{N}}$

**调度机制**

1. **FIFO（先进先出）调度**
   - 按照分组到来的次序发送
   - 队列满时的**丢弃策略**：
     - **尾部丢弃(tail drop)**：丢弃刚到达的分组
     - **优先权(priority)**：根据优先权丢失/移除分组
     - **随机(random)**：随机地丢弃/移除

2. **优先权调度**
   - 发送**最高优先权**的分组
   - 多类，不同类别有不同的优先权
   - 类别可能依赖于标记或者其他的头部字段（如IP源/目的地址、端口号、DS等）
   - 先传高优先级的队列中的分组，除非没有
   - 高（低）优先权中的分组传输次序：**FIFO**

3. **轮询(Round Robin, RR)调度**
   - 多类
   - 循环扫描不同类型的队列，发送完一类的一个分组，再发送下一个类的一个分组，循环所有类

4. **加权公平队列(Weighted Fair Queuing, WFQ)**
   - 一般化的Round Robin
   - 在一段时间内，每个队列得到的服务时间是：$\frac{W_i}{\sum W_j} \times t$，和权重成正比
   - 每个类在每一个循环中获得不同权重的服务量

**注意点**
- ⚠️ **缓存大小很关键**：太小 → 过度丢包；太大 → 过度延迟（bufferbloat/缓存膨胀）
- 💡 **理解技巧**：输出端口调度像医院分诊——FIFO是先来先治；优先权是急诊病人优先；WFQ是给不同科室保证时间配额
- 📋 **术语提醒**：HOL blocking = 头端阻塞；tail drop = 尾部丢弃；WFQ = 加权公平队列

## 🔑 核心要点总结

1. **路由器 = 控制平面 + 数据平面**：路由（软件，毫秒级）计算表；转发（硬件，纳秒级）移动分组
2. **输入端口**：物理接收 → 链路解封装 → 分布式转发表查找（基于目标或通用转发）
3. **最长前缀匹配**：TCAM硬件实现单时钟周期查找，不管表多大
4. **HOL阻塞**：输入队列队头分组等待繁忙输出端口，阻塞后面所有分组
5. **三种交换结构**：内存（单车道，CPU瓶颈）→ 总线（共享公路，带宽瓶颈）→ 纵横（多车道并发，60Gbps+）
6. **输出调度**：FIFO、优先权、轮询、WFQ——每种提供不同的QoS保证

## 📌 考试速记版

- **控制平面** = 路由算法（RIP/OSPF/BGP），生成路由表，软件，毫秒级
- **数据平面** = 转发，使用转发表，硬件，纳秒级
- **最长前缀匹配** = 最具体路由获胜；由**TCAM**实现
- **HOL阻塞** = 输入队列队头分组因输出端口竞争阻塞其他分组
- **交换结构**：内存（一次一个）、总线（一次一个，带宽受限）、纵横（多个并发）
- **缓存大小**：RFC 3439 = RTT × C；近期推荐 = RTT × C / √N
- **调度**：FIFO（公平无优先）、优先权（高优先级先）、RR（循环各类）、WFQ（按权重分配服务时间）

**记忆口诀**：
&gt; 路由慢在软件算，转发快在硬件干；
&gt; TCAM一周期找最长前缀；
&gt; HOL阻塞卡住整队输入线；
&gt; 内存总线纵横，吞吐量节节攀；
&gt; FIFO优先权轮询WFQ，选好你的队。
